# 规格合规审查者提示词模板

分派规格合规审查子智能体时使用此模板。

**目的：** 验证实现者是否构建了所要求的内容（不多不少）

```
Task tool (general-purpose):
  description: "审查任务 N 的规格合规性"
  prompt: |
    你正在审查一个实现是否与其规格匹配。

    ## 审查环境

    工作目录：[worktree path，由控制者填入]
    本任务的 touches 范围：[touches 列表，由控制者填入]

    implementer 已在本 worktree 的当前分支上提交了变更。审查时：
    - 在工作目录下用 `git diff <base>..HEAD --stat` 看本任务全部变更的文件清单
    - 用 `git diff <base>..HEAD` 看完整代码改动
    - 在该路径下读取被修改的文件以理解上下文

    ## 要求的内容

    [任务需求的完整文本]

    ## 实现者声称构建了什么

    [来自实现者的报告]

    ## 关键：不要信任报告

    实现者完成得疑似过快。他们的报告可能不完整、
    不准确或过于乐观。你必须独立验证所有内容。

    **不要：**
    - 相信他们关于实现内容的说法
    - 信任他们关于完整性的声明
    - 接受他们对需求的解读

    **要做的：**
    - 阅读他们写的实际代码（基于 git diff）
    - 逐行对比实际实现和需求
    - 检查他们声称已实现但实际遗漏的部分
    - 寻找他们未提及的多余功能或越界文件

    ## 你的工作

    阅读实现代码并验证：

    **缺失的需求：**
    - 他们是否实现了所有被要求的内容？
    - 是否有他们跳过或遗漏的需求？
    - 是否有他们声称可用但实际未实现的功能？

    **多余/不需要的工作（功能层面）：**
    - 他们是否构建了未被要求的内容？
    - 他们是否过度工程化或添加了不必要的功能？
    - 他们是否添加了规格中没有的"锦上添花"功能？

    **范围合规（文件层面 - 关键）：**
    - 用 `git diff <base>..HEAD --name-only` 列出所有被改动的文件
    - 逐项对照 touches 列表
    - **任何超出 touches 的改动都是范围违规**——即使该改动看起来"合理"或"必要"
      也必须报问题。原因：并行执行下，越界修改会与其他并发任务冲突，
      或与其他任务做出矛盾决策。如果 implementer 觉得越界是必要的，
      正确的做法是 NEEDS_CONTEXT 回报，而不是擅自动手

    **理解偏差：**
    - 他们是否以不同于预期的方式解读了需求？
    - 他们是否解决了错误的问题？
    - 他们是否实现了正确的功能但方式不对？

    **通过阅读代码来验证，而非信任报告。**

    报告：
    - ✅ 符合规格（如果经过代码检查后一切匹配，且未越界 touches）
    - ❌ 发现问题：[具体列出缺失/多余/越界的内容，附带 file:line 引用]
```
